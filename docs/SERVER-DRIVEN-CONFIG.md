# Server-driven configuration (всё на сервере)

Цель: приложение — «тонкий клиент». Никаких зашитых DNS/серверов и никакой
зависимости от чужих репозиториев. Вся конфигурация (резолверы, параметры
туннеля, операторы) приходит с нашего бэкенда; изменения применяются **без
пересборки приложения** — клиент подтягивает их при следующем подключении.

## Что где

| Данные | Источник | Endpoint |
|---|---|---|
| VLESS-серверы (режим «Скорость») | бэкенд | `GET /sub/{token}` |
| DNS-туннель: домен, ключ | бэкенд | `GET /api/dns-tunnel/verify/{token}` |
| Резолверы (оператор / noshape / yandex / fast / crowd / pool) | бэкенд | `GET /api/app-config` |
| Параметры туннеля (дупликация, воркеры, ARQ, MTU…) | бэкенд | `GET /api/app-config` → `tuning` |
| Crowd-статистика резолверов (приём от флота) | бэкенд | `POST /api/resolver-report` |
| Список операторов в Settings | из `/api/app-config` | — |

Бэкенд: `vpn-shop-bot` (отдельный репозиторий), `app/api/webhooks/dns_tunnel_api.py`,
крутится в контейнере `webhook` на бот-хосте; за `prismovpn.org/api/*` его
проксирует Caddy. Деплой = скопировать файл + `docker compose restart webhook`.

## 1. Резолверы: самоподдерживающийся per-network список

Серверная валидация резолверов **бесполезна** — резолвер, отвечающий из ДЦ,
может быть зашейплен/заблокирован у конкретного мобильного оператора под
цензурой. Поэтому отбор идёт **на устройстве** и **краудсорсингом**.

**Каталог (`/api/app-config`):**
- `carriers` — резолверы по операторам + MCC/MNC для авто-детекта (PLMN).
- `noshape` — универсальный тир «не шейпятся» (пробуется первым на любой сети).
- `yandex`, `fast` — фолбэки для Wi-Fi / неизвестного оператора.
- `crowd` (внутри каждого carrier) — резолверы, **подтверждённые флотом** как
  хорошо туннелирующие у этого оператора (см. п.2).
- `pool` — большой сырой пул кандидатов (курируемый + накраулленный, см. п.3).
  НЕ для прямого использования — только источник для on-device canary-проб.

**Порядок выбора (`ResolverListService.candidates`)**: crowd оператора →
резолверы оператора → noshape → yandex → остальные операторы (cap 40).

**On-device слой (iOS):**
- `ResolverHealthStore` — хранит рабочий набор per-network (ключ = PLMN или
  `wifi`) в `resolver-health-v2.json`. При реконнекте из той же сети рабочий
  набор переиспользуется мгновенно; при смене сети/устаревании — перепроба.
- `scanResolvers` пробует `candidates()` на текущей сети и сохраняет живые.
- Во время сессии `ClientViewModel.startResolverHealthPruning` каждые 30с
  читает пассивную статистику движка (`GetResolverStats`) и **выкидывает**
  явных аутсайдеров (высокая потеря / invalid) из рабочего набора.

> **Урок:** широкие сырые пулы (типа `verified.txt` на 43k IP) **роняют**
> скорость — большинство отвечают на DNS, но не туннелируют на конкретной
> мобильной сети, и балансировщик тратит пакеты на мёртвые пути. Помогают
> только tunnel-проверенные, пооператорные и non-shaping резолверы.

## 2. Crowd-sourced резолверы (флот сам поддерживает списки)

Каждое устройство меряет РЕАЛЬНОЕ туннельное здоровье резолверов на своей
сети и анонимно шлёт сводку на бэкенд. Бэкенд агрегирует по оператору/региону
и возвращает ранжированные списки — выигрывает весь флот, без ручной валидации.

- **App → `POST /api/resolver-report`** (iOS, раз в ~5 мин в pruning-таске):
  `{plmn, samples:[{resolver, sent, acked, lost, rtt_ms}]}`. **Анонимно** —
  ни токена, ни user id, только PLMN + счётчики.
- **Бэкенд (`ResolverCrowdStore`)**: EMA success-ratio + RTT по
  `(scope, resolver)`, scope = id оператора или `region:<MCC>`. Старые записи
  выветриваются (`MAX_AGE`). Персист в файл (`RESOLVER_CROWD_PATH`).
- **Обратно в `/api/app-config`**: списки резолверов оператора
  **переранжируются** по crowd-данным (проверенные — вперёд), плюс поле
  `crowd` с лучшими резолверами оператора (может включать те, которых нет в
  статическом каталоге — они доказанно туннелируют у этого оператора).

> Репортинг — пока только iOS (нужна per-resolver статистика движка). Android
> потребляет crowd-ранжированные списки, но не репортит (см. п.5).

## 3. Бэкенд-краулер + canary-тестирование

- **Краулер** (`app.py`, фоновая задача, раз в сутки): тянет публичные RU
  резолверы (`public-dns.info`) в canary-пул (`RESOLVER_CANARY_PATH`, cap 400)
  и отдаёт их в `/api/app-config` → `pool`. Выключается `RESOLVER_CRAWLER_ENABLED=0`.
- **Canary на устройстве (iOS)**: `scanResolvers` на ~каждом 4-м скане пробует
  случайную выборку из `pool` и добавляет до 3 достижимых в рабочий набор (в
  пределах cap 40). Движок измеряет их под туннелем → crowd-report продвигает
  хорошие, плохие отсеиваются. Замкнутый цикл: краул → канарейка флотом →
  агрегация → ранжированные списки всем. Без риска «широкого пула» (кэп жёсткий).

## 4. Server-driven тюнинг туннеля (per-operator)

Параметры движка (дупликация пакетов, воркеры, ARQ-окно, MTU, батч, стратегия
балансировки) теперь приходят с сервера — крутятся вживую, глобально и по
оператору, без пересборки.

- **Бэкенд**: `TUNNEL_TUNING_DEFAULT` (база для всех + Wi-Fi/неизвестные) и
  `TUNNEL_TUNING_BY_CARRIER` (оверрайды по id оператора). Отдаётся в
  `/api/app-config` → `tuning: {default, by_carrier}`.
- **iOS**: `AppConfigCatalog.Tuning` + `resolvedTuning(plmn:pinnedCarrierID:)`
  мёржит default с оверрайдом оператора; `ConfigBuilder` подставляет значения
  в TOML (фолбэк на встроенные дефолты, если сервер ничего не дал).

**Текущий проверенный профиль (дефолт для всех, валидирован на МТС/Yota):**

| Параметр | Значение | Зачем |
|---|---|---|
| `packet_duplication_count` | 2 | дупликация = ×N к upstream DNS-запросам; 5 душило аплинк и ловило рейт-лимит оператора |
| `setup_packet_duplication_count` | 2 | меньше флуда на создании лавины стримов (Telegram открывает ~150) |
| `rx_tx_workers` / `tunnel_process_workers` | 8 / 12 | переживать всплески стримов |
| `max_packets_per_batch` | 12 | менее «взрывная» отправка |
| `arq_window_size` | 1000 | **ключевое**: малое окно = мало bufferbloat = нет «затыка» после пика + выше реальный download |
| `max_upload_mtu` | 200 | потолок аплоада (реально сеть даёт ~137 байт/запрос) |
| `max_download_mtu` | 4000 | верхняя граница пробинга download MTU |

> **Главный инсайт по скорости:** узкое место — не «медленные резолверы», а
> **bufferbloat**. Снижение ARQ-окна и дупликации убрало «удушение после
> нагрузки» И подняло download (на Yota 14→18 Мбит/с). Upload (~0.4 Мбит/с) —
> физический потолок DNS-туннеля (кодируется в имени DNS-запроса), сильно не
> растёт, но медиа долетает.

Чтобы завести оператору отдельный профиль (например, больше избыточности на
лоссовой сети) — добавить запись в `TUNNEL_TUNING_BY_CARRIER`, редеплой.

## 5. Применение на клиентах

- **iOS `AppConfigService.refresh()`** ходит на `/api/app-config` с
  `reloadIgnoringLocalCacheData` — игнорирует локальный HTTP-кэш, всегда тянет
  свежий конфиг при запуске.
- **`ClientViewModel.start()`** ждёт `await AppConfigService.shared.refresh()`
  **перед** стартом движка — гарантирует, что тюнинг/резолверы оператора
  применяются на этом подключении (а не проигрывают гонку фоновому refresh).
  Без этого фикса тюнинг не доезжал и движок стартовал на дефолте.
- Ответ кешируется на эдже `max-age=600`. Версия — `APP_CONFIG_VERSION`.

**Android (`prismo-dns-app`):**
- `AppConfigManager` — фетч `/api/app-config`, кэш, выбор резолверов по PLMN
  (`TelephonyManager`, без доп. разрешений), bundled-fallback офлайн.
- `DnsTunnelVpnService.startTunnel` тянет каталог и передаёт пооператорные
  резолверы в `DnsTunnelProcess` (раньше было 4 захардкоженных IP).
- Потребляет резолверы; crowd-репортинг и применение `tuning` — TODO (нужна
  per-resolver статистика из бинаря `mdns_client`).

## Диагностика по логам движка

В логах клиента видно, применился ли тюнинг и что происходит под нагрузкой:
- `Note: Each packet will be sent N times` — N = `packet_duplication_count`
  (должно быть 2, не 5).
- `Async Runtime Initialized: 8 RX/TX Workers, 12 Processors` — воркеры из тюнинга.
- `Selected Synced Upload/Download MTU` — итоговые MTU после пробинга.
- `DNS Resolver disabled (100% Loss)` — авто-дизейбл (есть пол `minActive≥2`,
  все резолверы разом не выключаются).
- Резкий **обрыв лога без ошибок** под нагрузкой ⇒ вероятно iOS усыпил процесс
  приложения (другой класс проблемы: keep-alive / Network Extension).

## Env-флаги бэкенда (резолверы/тюнинг)

| Переменная | Дефолт | Назначение |
|---|---|---|
| `RESOLVER_CROWD_PATH` | temp | персист crowd-агрегата |
| `RESOLVER_CANARY_PATH` | temp | персист накраулленного пула |
| `RESOLVER_CRAWLER_ENABLED` | `1` | вкл/выкл краулер |
| `RESOLVER_CRAWL_INTERVAL` | `86400` | интервал краула, сек |

> Рекомендуется задать `RESOLVER_*_PATH` на постоянный диск (например
> `/var/lib/prismo/...`), иначе данные обнулятся при пересоздании контейнера
> (рестарт переживают, rebuild — нет). Данные самовосстанавливаются из
> репортов флота + суточного краула.
